# 衣来 MVP 试点与会员策略建议

## 1. 当前实现结论

### 衣物识别

当前版本没有真实图片识别能力。此前 `utils/recognizer.js` 使用图片路径生成 seed，再伪随机填充品类、颜色、材质和风格，这会让用户误以为系统在“瞎识别”。

已调整为：

- 上传图片只保存图片。
- 用户手动选择品类、颜色、季节等信息。
- 系统只基于用户已选择的品类/颜色做季节、主风格、风格标签建议。
- `recognizer.js` 仅保留占位返回，不再伪造识别结果。
- 单件录入页已移除“自动识别/自动预填”文案。

后续如需自动识别，必须接入真实模型，并在 UI 中区分“识别建议，可修改”，不要把建议当成确定结果。

### 天气

当前天气逻辑：

- 推荐页会先尝试 `wx.getLocation` 获取经纬度。
- 如果关闭定位或定位失败，使用设置页城市。
- `config/qweather.js` 已配置和风天气 API Key，MVP 试点阶段采用前端 API KEY 直连。
- 定位成功时，直接用经纬度请求实时天气，避免多调用一次 GeoAPI。
- 关闭定位或定位失败时，使用设置页城市，先调用 GeoAPI 获取 LocationID，再调用实时天气。
- API Key 为空、网络失败或接口异常时，回退到“天气不可用 / 默认天气”，避免误导用户以为北京天气是真实结果。
- 当前测试 Key 已改用和风天气控制台提供的专属 API Host：`https://my3dnabwq6.re.qweatherapi.com`。如果请求失败，优先检查该凭据是否启用了 `GeoAPI` 与 `天气预报`，以及小程序预览/上线环境的 request 合法域名配置。

结论：当前具备实时天气接入条件；需要在微信开发者工具和真机中验证定位授权、request 合法域名和和风天气控制台调用统计。

## 2. 免费试点阶段建议限制

试点阶段不要限制得太狠，否则会影响用户验证核心价值。建议只限制会带来成本、复杂度或会员转化空间的能力。

### 免费版建议

- 衣物数量：最多 30 件。
- 每次批量导入：最多 9 张图片。
- 穿搭日历保存：最近 30 天。
- 推荐刷新/换一套：每日 20 次。
- 场景数量：保留基础场景 4 个，上班、约会、休闲、运动。
- 图片云存储：暂不开放或限制总容量。
- 数据导出/备份：不开放。
- 高级统计：不开放。

### 会员版建议

- 衣物数量：300 件或不限。
- 每次批量导入：最多 50 张图片。
- 穿搭日历：永久保存。
- 推荐刷新：不限或高额度。
- 高级场景：婚礼、面试、旅行、晚宴、见家长等。
- 衣物使用统计：闲置提醒、使用次数、单次穿着成本。
- 云同步：多设备同步、图片云存储。
- 高级推荐：加入风格偏好、历史避重、天气指数、色彩搭配解释。

### 不建议一开始收费限制的能力

- 基础推荐不要限制，否则用户感受不到产品价值。
- 添加第一批衣服不要限制太低，10 件以内很难形成好推荐。
- 基础日历不要完全关闭，否则“确认穿搭”的闭环会断。

## 3. 推荐优先做的会员权限点

优先级从高到低：

1. 衣物数量上限
2. 批量导入数量上限
3. 历史穿搭保存时长
4. 云同步/云备份
5. 高级统计
6. 高级场景和高级推荐解释
7. 图片识别或 AI 自动打标

最推荐的商业化包装：

- 免费版：轻量衣橱，适合试用。
- Pro 版：完整衣橱、批量导入、长期日历、云同步。

## 4. 批量上传录入产品逻辑

### 核心原则

批量上传不应该批量“自动乱填”，而是批量创建草稿，再让用户快速补信息。

推荐流程：

1. 用户点击“批量添加”。
2. 从相册选择多张图片。
3. 系统生成“待整理”草稿列表。
4. 每张图片先只保存图片和默认空字段。
5. 用户进入批量整理页，以卡片方式逐张补充信息。
6. 支持批量设置共用字段，例如季节、品类、颜色。
7. 用户点击“保存全部”，通过校验的衣物入库，未完成的保留草稿。

### 交互结构

批量整理页建议包含：

- 顶部进度：`已完成 3 / 共 9 件`
- 图片横向缩略图列表
- 当前大图预览
- 必填字段：品类、细分品类、颜色、名称、季节
- 批量操作：
  - 全部设为春夏
  - 全部设为秋冬
  - 全部设为上装/下装/鞋子
  - 跳过当前
  - 保存当前并下一件
- 底部主按钮：保存全部可用项

### 免费/会员限制

- 免费：一次最多 9 张。
- 会员：一次最多 50 张。
- 免费衣物总数达到上限时，仍可选择图片，但保存时提示升级或删除旧衣物。

当前已实现：

- 免费版衣物数量上限：30 件。
- 衣橱页展示当前免费额度。
- 单件保存前检查上限。
- 批量导入入口。
- 批量导入一次最多选择 9 张，且不能超过剩余衣物额度。
- 批量导入采用“草稿整理”模式，不做图片内容猜测。
- 批量整理页支持逐张编辑、保存当前并下一件、保存全部可用项。

### 不接 AI 识别时的提效方式

- 根据用户先选的“大品类”自动展开细品类。
- 根据细品类推断季节和风格。
- 支持对多张图片批量套用同一季节。
- 保存当前后自动跳下一张，减少返回列表操作。

## 5. 天气 API 接入建议

建议使用和风天气 QWeather。

当前代码已按 QWeather 写了接口：

- GeoAPI 城市/经纬度查询：`/geo/v2/city/lookup`
- 实时天气：`/v7/weather/now`
- 配置文件：`config/qweather.js`
- 当前凭据只需要启用 `GeoAPI` 和 `天气预报`。

### 获取方式

1. 注册和风天气开发者账号。
2. 进入控制台，创建项目。
3. 在项目中添加凭据。
4. MVP 阶段可以先创建 API KEY 凭据。
5. 把 Key 填入 `config/qweather.js` 的 `apiKey`。当前 MVP 测试 Key 已写入本地配置。

### 免费策略

和风天气官方说明提供免费订阅/免费请求额度，超出后可按量计费。试点阶段建议：

- 只请求实时天气，不请求多日预报、空气质量等扩展能力。
- 对同一城市/经纬度结果做短时缓存，例如 30-60 分钟。
- 用户频繁切换页面时不要重复请求。
- 后续商业化后再扩展天气指数、降雨提醒等会员功能。

### 安全建议

微信小程序前端直接放 API Key 有泄露风险。MVP 内测可以先这样跑通，但正式上线建议：

- 通过云函数代理请求天气。
- API Key/JWT 私钥只放云函数环境变量。
- 小程序只请求自己的云函数。

云函数代理含义：

- 当前直连：小程序前端 -> 和风天气。
- 云函数代理：小程序前端 -> 微信云函数 -> 和风天气。

云函数代理目的：

- 小程序代码包可能被反编译，前端 Key 容易泄露。
- Key 泄露后，别人可以拿你的 Key 调接口，消耗免费额度或产生费用。
- 云函数可以把 Key 放在环境变量里，前端看不到。
- 云函数可以做限流和缓存，例如同一城市 30-60 分钟内只请求一次和风天气。
- 后续可以在云函数里升级 JWT 鉴权，不影响小程序前端调用方式。
